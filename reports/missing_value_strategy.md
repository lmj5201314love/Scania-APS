# Scania APS 缺失值处理策略

## 文档目的

本文档用于记录 Scania APS 项目在 Day 2 数据质量分析之后形成的缺失值处理思路。当前阶段只制定策略，不执行缺失值填充、不删除字段、不生成最终建模数据，也不训练模型。

## 已观察到的缺失值特点

Day 2 分析显示，Scania APS 原始数据不是少量随机缺失，而是具有明显工业数据特征的结构性缺失：

- 训练集整体单元格缺失率约为 8.28%。
- 测试集整体单元格缺失率约为 8.36%。
- train/test 均有 169 列存在缺失。
- train/test 缺失率分层分布一致。
- train/test 缺失模式整体接近，最大缺失率差异约为 0.55 个百分点。
- pos/neg 样本之间存在明显不同的缺失模式。

这些观察说明，缺失值本身是项目核心数据质量问题之一，后续处理不能简单粗暴。

## 高缺失字段

Day 2 识别出的高缺失字段包括：

- `br_000`
- `bq_000`
- `bp_000`
- `bo_000`
- `ab_000`
- `cr_000`
- `bn_000`
- `bm_000`

这些字段缺失率较高，但不能在当前阶段直接删除。原因是：在工业传感器数据中，高缺失可能代表设备状态、采集条件或系统运行模式，而不一定代表字段无价值。

## pos/neg 缺失模式差异

Day 2 发现部分字段在正负样本之间的缺失率差异很大。例如：

- `br_000`、`bq_000`、`bp_000`、`bo_000`、`bn_000`、`bm_000` 在 neg 中缺失率明显高于 pos。
- `ak_000`、`ca_000`、`di_000`、`dj_000`、`dk_000`、`eb_000` 在 pos 中缺失率明显高于 neg。

当前只能将这一点记录为“后续值得评估的缺失模式差异”，不能直接断言缺失本身一定可以作为故障特征。

## 后续处理原则

1. 不修改 `data/raw/` 下的原始数据。
2. 不在测试集上拟合任何处理规则。
3. 所有填充值、缺失指示规则、字段筛选规则只能在训练集或训练集内部验证集上确定。
4. 高缺失字段不直接删除，必须通过实验比较后决定。
5. 任何缺失处理策略都要结合 recall、F2、PR-AUC 和业务成本，而不是只看 accuracy。

## 建模阶段可比较的策略

后续 Day 4 之后可以逐步比较以下策略：

### 方案 A：保留字段 + 简单填充

在训练集上计算数值字段的中位数，并将同一填充值应用到验证集和测试集。

适用场景：建立 baseline，观察最基础的缺失处理效果。

注意事项：中位数只能从训练集计算，不能使用测试集。

### 方案 B：保留字段 + 缺失指示变量

对缺失模式差异明显的字段增加 `is_missing` 指示变量，再进行填充。

适用场景：评估缺失模式本身是否有预测价值。

注意事项：当前不能预先断言该方案一定更好，必须通过验证集指标和业务成本比较。

### 方案 C：模型原生处理缺失

对于 XGBoost 等能够处理缺失值方向的模型，可以保留 `NaN`，让模型学习缺失方向。

适用场景：提升模型阶段。

注意事项：仍然需要防止测试集泄漏，并对结果进行业务解释。

### 方案 D：高缺失字段敏感性实验

比较“保留高缺失字段”和“删除高缺失字段”的模型表现差异。

适用场景：判断 `br_000`、`bq_000`、`bp_000` 等高缺失字段是否实际贡献有效信息。

注意事项：删除规则必须基于训练集确定，并且不能只凭缺失率做最终决定。

## 当前阶段不执行的操作

Day 3 不执行以下操作：

- 不做 median 填充。
- 不删除高缺失字段。
- 不生成最终建模数据。
- 不训练 Logistic Regression、Random Forest 或 XGBoost。
- 不计算模型 precision、recall 或阈值成本曲线。
- 不进行风险分层建模。

## 与 SQL 分析的关系

Day 3 的 SQL 脚本用于复核缺失率分布、高缺失字段、train/test 缺失差异和 pos/neg 缺失差异。SQL 复核完成后，可以将缺失分析结果作为后续建模策略设计的依据，但不能替代建模阶段的验证实验。

## 后续建议

Day 4 进入 baseline 前，应先明确训练集内部验证方案。缺失值处理策略应从最简单、最可解释的方案开始，再逐步加入缺失指示变量或模型原生缺失处理，并始终用业务成本意识解释结果。
